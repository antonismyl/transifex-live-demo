Scaling our event pipeline to a billion events a day.

Marcus Lindqvist · Feb 24, 2026 · 9 min read

A year ago, Lumen's event pipeline peaked somewhere around 40,000 events per second. Today we sit comfortably above 12,000 per second on an average Tuesday, and our weekly peaks cross a billion events per day. The shape of that growth forced us to revisit almost every piece of the ingestion path.

The old architecture

For the first two years, ingestion looked like most textbook event pipelines: an HTTP front-door written in Go, a Kafka buffer, a set of stateless enrichment workers, and a columnar store (ClickHouse) as the final destination. It worked. Until it didn't.

The failure mode wasn't capacity — Kafka scales much further than we needed. It was tail latency. Customer SDKs retry aggressively; when enrichment workers fell behind, the Kafka lag alarms would fire and the on-call engineer would page. The fix was almost always horizontal — add partitions, add workers — but the coupling was uncomfortable.

What we changed

We made three changes, in order of increasing scariness:

  1. Introduced an in-process buffer at the HTTP front-door with explicit back-pressure. Overloaded workers return 429, the SDK retries with jitter, the caller is never blocked.
  2. Split the ingestion path into a hot tier (real-time dashboards) and a warm tier (everything else). Hot-tier events skip Kafka entirely.
  3. Replaced the Kafka-based warm tier with a custom append-only log on NVMe SSDs, fronted by a small Raft group.

That last point is the one people want to ask about, so let me save you the trouble: yes, we considered just running a bigger Kafka. Yes, we considered Pulsar and Redpanda. We ran both in shadow for about six weeks. They were good. They weren't right for us — our workload looks nothing like a typical log-streaming workload, because we only ever replay events within a 48-hour window.

If you're reaching for a distributed log and you only ever read from the tail, you probably don't want a distributed log.

A concrete example

Here's the shape of the new hot-tier handler, simplified to the point of dishonesty:

func (s *Server) Ingest(w http.ResponseWriter, r *http.Request) {
  evt, err := decode(r.Body)
  if err != nil {
    http.Error(w, err.Error(), 400)
    return
  }
  select {
  case s.hot <- evt:
    w.WriteHeader(202)
  default:
    // Channel full — ask the client to retry.
    w.WriteHeader(429)
  }
}

The interesting part isn't the code. It's what isn't there: no locks, no fanout, no retries. When the channel is full, we say so. Everything downstream of the channel runs at a predictable pace.

What we're watching next

We're not done. The warm tier still leans heavily on ClickHouse, and we're tracking a few new options we'd like to evaluate later this year. We'll post again when we have something worth saying.

If pipelines like this are your idea of a good time — we're hiring. See you in the Kafka-shaped hole in the ground.
